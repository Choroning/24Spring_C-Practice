## 1. 프로그래밍 개념

### 1.1 소프트웨어와 프로그램

**하드웨어**는 컴퓨터의 물리적 구성 요소(CPU, 메모리, 저장장치, 주변장치 등)이다. **소프트웨어**는 하드웨어에게 무엇을 할지 지시하는 프로그램의 집합이다.

```
컴퓨터 시스템
├── 하드웨어
│   ├── CPU (중앙 처리 장치)
│   ├── 메모리 (RAM)
│   ├── 저장장치 (HDD/SSD)
│   └── 입출력 장치 (키보드, 모니터 등)
└── 소프트웨어
    ├── 시스템 소프트웨어 (운영체제, 드라이버, 유틸리티)
    └── 응용 소프트웨어 (워드 프로세서, 게임 등)
```

**프로그램**은 특정 작업을 수행하기 위해 프로그래밍 언어로 작성된 명령어의 집합이다. **프로그래밍**은 이러한 명령어를 설계하고 작성하는 과정이다.

> **핵심:** 소프트웨어가 없으면 하드웨어는 단순한 전자 부품 모음에 불과하다. 소프트웨어가 하드웨어에 목적을 부여한다.

### 1.2 프로그래밍 언어

**프로그래밍 언어**는 컴퓨터에 명령을 전달하기 위한 형식 언어이다. 기계 하드웨어와의 근접도에 따라 분류한다.

| 수준 | 언어 | 예시 | 가독성 | 이식성 |
|:-----|:----|:-----|:-----:|:-----:|
| 저급 | 기계어 | `10110000 01100001` | 매우 낮음 | 없음 |
| 저급 | 어셈블리어 | `MOV AL, 61h` | 낮음 | 없음 |
| 고급 | C, Java, Python | `printf("Hello");` | 높음 | 높음 |

**기계어** (1세대):
- CPU가 직접 실행하는 이진 코드(`0`과 `1`).
- 각 명령어가 특정 CPU 연산에 대응한다.
- CPU 아키텍처마다 다르므로 완전히 **기계 종속적**이다.

**어셈블리어** (2세대):
- 이진 코드 대신 사람이 읽을 수 있는 **니모닉**(예: `MOV`, `ADD`, `JMP`)을 사용한다.
- 기계어와 1:1 대응한다.
- **어셈블러**가 기계어로 번역한다.
- 여전히 기계 종속적 — x86용 어셈블리는 ARM에서 실행할 수 없다.

**고급 언어** (3세대 이상):
- 자연어에 가까운 문법(`if`, `while`, `return`)을 사용한다.
- 하나의 고급 문장이 여러 기계어 명령으로 번역될 수 있다.
- **컴파일러** 또는 **인터프리터**가 기계어로 번역한다.
- 일반적으로 **기계 독립적** — 동일한 소스 코드가 다른 플랫폼에서 실행 가능하다.

```
소스 코드 (고급 언어)
        │
        ▼
   ┌──────────┐
   │  컴파일러  │  ← 전체 프로그램을 한 번에 번역
   └──────────┘
        │
        ▼
기계어 코드 (이진수)
        │
        ▼
   ┌──────────┐
   │   CPU    │  ← 실행
   └──────────┘
```

> **팁:** **컴파일러**는 전체 소스 코드를 한꺼번에 기계어로 번역한 뒤 실행한다 (C, C++, Java). **인터프리터**는 실행 시점에 한 줄씩 번역하며 실행한다 (Python, JavaScript).

### 1.3 기계 종속 vs 기계 독립

| 특성 | 기계 종속 | 기계 독립 |
|:----|:--------:|:--------:|
| 특정 CPU에 종속? | 예 | 아니오 |
| 예시 | 기계어, 어셈블리어 | C, Java, Python |
| 이식성 | 다른 아키텍처로 이식 불가 | 동일 소스로 다른 시스템에서 실행 가능 |
| 성능 | 최대 (하드웨어 직접 제어) | 약간 낮음 (번역 오버헤드) |
| 개발 속도 | 매우 느림 | 빠름 |

기계어와 어셈블리어는 **기계 종속적**이다 — 특정 CPU 아키텍처용으로 작성된 코드는 다른 아키텍처에서 재작성 없이 실행할 수 없다.

고급 언어는 **기계 독립적**이다 — 동일한 C 소스 코드를 각 시스템에 맞는 컴파일러로 컴파일하면 다양한 플랫폼(Windows, Linux, macOS)에서 실행할 수 있다.

> **핵심:** "기계 독립"이 이식에 노력이 전혀 필요 없다는 뜻은 아니다. 플랫폼 고유 기능(예: 시스템 호출)은 여전히 다를 수 있다. 그러나 핵심 언어 문법과 로직은 동일하게 유지된다.

## 2. C 언어

### 2.1 역사

C 언어는 **1972년** Bell 연구소에서 **Dennis Ritchie**가 **Ken Thompson**이 개발한 B 언어를 기반으로 만들었다. 주된 목적은 UNIX 운영체제를 고급 언어로 다시 작성하는 것이었다.

```
C 언어의 역사
──────────────────────────────────────────────────────────────
1969    B 언어 (Ken Thompson) — PDP-7에서 초기 UNIX 개발용
1972    C 언어 (Dennis Ritchie) — Bell Labs, PDP-11에서 UNIX 개발용
1978    K&R C — "The C Programming Language" 서적 출간 (Kernighan & Ritchie)
1989    ANSI C (C89) — 최초의 공식 표준 (ANSI X3.159-1989)
1990    C90 — ISO에서 ANSI C 채택 (ISO/IEC 9899:1990)
1999    C99 — 인라인 함수, 가변 길이 배열, // 주석 도입
2011    C11 — 멀티스레딩, _Generic, 경계 검사 도입
2018    C18 — C11 버그 수정 (새로운 기능 없음)
2023    C23 — 최신 표준 (nullptr, #embed, auto 타입 추론)
──────────────────────────────────────────────────────────────
```

> **핵심:** "ANSI C" 또는 "표준 C"라고 하면 보통 C89/C90을 의미한다 — 가장 널리 지원되는 기본 표준이다. 대부분의 현대 컴파일러는 최소 C99를 지원한다.

### 2.2 특징

C 언어가 50년 넘게 사용되는 이유는 독특한 강점의 조합에 있다:

| 특징 | 설명 |
|:----|:----|
| **간결한 문법** | 적은 수의 키워드 (C89 기준 ~32개); 최소한의 언어 설계 |
| **유연성** | 제약이 적음 — 프로그래머가 세밀하게 제어 가능 |
| **고성능** | 효율적인 기계어로 컴파일됨; 런타임 오버헤드 최소 |
| **이식성** | 표준 C 코드는 C 컴파일러가 있는 거의 모든 플랫폼에서 컴파일 가능 |
| **저수준 지원** | 포인터를 통한 직접 메모리 접근; 비트 연산; 하드웨어 제어 |
| **효율성** | 작은 메모리 사용량; 자원이 제한된 환경에 적합 |
| **현대 언어의 뿌리** | C++, Java, C#, Python, Go, Rust 등이 C의 문법과 개념에서 영향을 받음 |

```c
#include <stdio.h>

int main(void) {
    printf("Hello, World!\n");
    return 0;
}
```

> **팁:** C는 최대한의 제어권을 제공하지만, 그만큼 책임도 따른다. 가비지 컬렉터, 배열 범위 검사, 내장 예외 처리가 없으므로 프로그래머가 모든 것을 직접 관리해야 한다.

### 2.3 활용 분야

C는 성능, 이식성, 저수준 하드웨어 접근이 중요한 모든 곳에서 사용된다.

| 분야 | 예시 |
|:----|:----|
| 운영체제 | UNIX, Linux 커널, Windows 커널, macOS 커널 |
| 컴파일러 | GCC, Clang, MSVC |
| 데이터베이스 | MySQL, PostgreSQL, SQLite |
| 임베디드 시스템 | 마이크로컨트롤러, IoT 디바이스, 자동차 ECU |
| 게임 엔진 | Unreal Engine (C/C++), id Tech 엔진 |
| 네트워크 | TCP/IP 스택, Apache, Nginx |
| 과학 계산 | MATLAB 내부, R 내부, 수치 라이브러리 |

> **핵심:** Linux 커널(3천만 줄 이상의 코드)은 거의 전부 C로 작성되어 있다. 이는 C가 작은 임베디드 시스템부터 거대하고 복잡한 소프트웨어까지 확장 가능함을 보여준다.

### 2.4 C를 배워야 하는 이유

1. **기초 체력** — C는 컴퓨터가 실제로 어떻게 작동하는지를 가르쳐 준다: 메모리 구조, 포인터, 스택과 힙, 수동 자원 관리. 이 개념들은 어디서나 적용된다.

2. **공용어(Lingua Franca)** — C는 시스템 프로그래밍의 공용어이다. OS API, 하드웨어 인터페이스, 수많은 라이브러리가 C 인터페이스를 제공한다. C를 이해하면 전체 소프트웨어 생태계와 소통할 수 있다.

3. **다른 언어의 기반** — C를 배우면 C++, Java, C#, Go, Rust 등을 배우기가 훨씬 쉬워진다. 이 언어들은 C의 문법과 의미론을 상당 부분 차용했기 때문이다.

```
C (1972)
├── C++ (1979) — OOP, 템플릿, STL 추가
├── Objective-C (1984) — Smalltalk 스타일 OOP 추가
├── Java (1995) — C 유사 문법, 가비지 컬렉션, JVM
├── C# (2000) — Microsoft의 Java 대항마
├── Go (2009) — C의 단순함 + 동시성
└── Rust (2010) — C급 성능 + 메모리 안전성
```

> **팁:** Python이나 JavaScript를 주로 사용할 계획이더라도, C를 알면 "내부에서 무슨 일이 일어나는지" — 메모리 할당, CPU 수준의 동작, 특정 연산이 왜 빠르거나 느린지 — 를 깊이 이해할 수 있다.

## 3. C 프로그램 개발

### 3.1 개발 과정

C 프로그램을 만드는 전체 과정은 네 가지 주요 단계를 따른다:

```
┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
│   1. 설계     │───▶│  2. 코드 작성  │───▶│   3. 컴파일    │───▶│  4. 실행      │
│   (알고리즘)   │    │  (소스 코드)   │    │  (빌드)       │    │  (실행 & 검증)  │
└──────────────┘    └──────────────┘    └──────────────┘    └──────────────┘
                                               │
                                            오류 발생?
                                               │
                                               ▼
                                        ┌──────────────┐
                                        │   5. 디버깅    │
                                        │   (오류 수정)  │
                                        └──────────────┘
                                               │
                                               └──── 2단계 또는 3단계로 돌아감
```

| 단계 | 활동 | 산출물 |
|:----|:----|:------|
| 설계 | 문제 분석; 알고리즘과 자료구조 설계 | 의사 코드, 순서도 |
| 코드 작성 | 텍스트 편집기 또는 IDE에서 소스 코드 작성 | `.c` 소스 파일 |
| 컴파일 | 소스 코드를 기계어로 번역; 빌드 오류 수정 | 실행 파일 (바이너리) |
| 실행 | 프로그램 실행; 올바른 출력 확인 | 프로그램 출력 |
| 디버깅 | 오류 식별 및 수정; 컴파일-실행 반복 | 수정된 소스 코드 |

### 3.2 빌드 과정

C 프로그램을 "컴파일"할 때, 내부적으로 여러 단계가 진행된다. 전체 빌드 파이프라인은 다음과 같다:

```
  hello.c          stdio.h
     │                │
     ▼                │
┌────────────┐        │
│  전처리기    │◀───────┘   (#include, #define 확장)
└────────────┘
     │
     ▼
  hello.i          (전처리된 소스 — 모든 매크로 확장됨)
     │
     ▼
┌────────────┐
│   컴파일러   │             (C 코드 → 어셈블리 코드)
└────────────┘
     │
     ▼
  hello.s          (어셈블리 소스)
     │
     ▼
┌────────────┐
│   어셈블러   │             (어셈블리 → 오브젝트 코드)
└────────────┘
     │
     ▼
  hello.o          (오브젝트 파일 — 기계어, 아직 실행 불가)
     │
     ▼
┌────────────┐
│    링커     │◀─── 라이브러리 파일 (libc.a 등)
└────────────┘
     │
     ▼
  hello.exe        (최종 실행 파일)
```

| 단계 | 입력 | 출력 | 하는 일 |
|:----|:----|:----|:-------|
| **전처리기** | `.c` 소스 | `.i` 전처리 파일 | `#include`, `#define`, `#ifdef` 확장; 주석 제거 |
| **컴파일러** | `.i` 전처리 파일 | `.s` 어셈블리 | C를 어셈블리로 번역; 문법 및 타입 검사 |
| **어셈블러** | `.s` 어셈블리 | `.o` 오브젝트 | 어셈블리를 이진 기계어로 변환 |
| **링커** | `.o` 오브젝트 + 라이브러리 | 실행 파일 | 오브젝트 파일 결합; 외부 참조 해결 (예: `printf`) |

> **핵심:** 전처리기는 실제 컴파일 **이전에** 실행된다. 텍스트 대치를 수행하며, C 문법을 이해하지 못한다. 이것이 `#define` 매크로가 미묘한 버그를 일으킬 수 있는 이유이다.

실제로는 컴파일러 드라이버(예: `gcc`)가 모든 단계를 자동으로 처리한다:

```
gcc hello.c -o hello      ← 전처리기 + 컴파일러 + 어셈블러 + 링커
gcc -E hello.c             ← 전처리기만 실행 (.i 출력)
gcc -S hello.c             ← 전처리기 + 컴파일러만 (.s 출력)
gcc -c hello.c             ← 전처리기 + 컴파일러 + 어셈블러 (.o 출력)
```

### 3.3 오류의 종류

C 프로그램의 오류는 **감지 시점**에 따라 세 가지로 분류된다:

| 오류 유형 | 감지 시점 | 원인 | 예시 |
|:---------|:---------|:----|:----|
| **컴파일 오류** | 컴파일 시 | 문법 오류, 타입 불일치, 미선언 변수 | `;` 누락, 잘못된 타입 |
| **링크 오류** | 링킹 시 | 함수 정의 누락, 해결되지 않은 심볼 | 존재하지 않는 함수 호출 |
| **런타임 오류** | 실행 시 | 논리 오류, 불법 연산 | 0으로 나누기, 널 포인터 접근 |

**컴파일 오류** (문법 오류):
```c
#include <stdio.h>

int main(void) {
    printf("Hello\n")   // ← 세미콜론 누락 → 컴파일 오류
    return 0;
}
```

**링크 오류:**
```c
// main.c
void myFunction(void);  // 선언만 하고 어디서도 정의하지 않음

int main(void) {
    myFunction();        // ← 링커가 정의를 찾을 수 없음 → 링크 오류
    return 0;
}
```

**런타임 오류** (논리 오류):
```c
#include <stdio.h>

int main(void) {
    int a = 10, b = 0;
    printf("%d\n", a / b);  // ← 0으로 나누기 → 런타임 오류 (프로그램 비정상 종료)
    return 0;
}
```

> **경고:** 컴파일 오류는 가장 고치기 쉽다 — 컴파일러가 줄 번호와 원인을 알려준다. 런타임 오류가 가장 어렵다 — 프로그램이 컴파일되고 실행되지만, 잘못된 결과를 출력하거나 예기치 않게 비정상 종료된다.

### 3.4 디버깅

**디버깅**은 프로그램의 오류(버그)를 찾아서 수정하는 과정이다. 이 용어는 1947년 Grace Hopper 팀이 Harvard Mark II 컴퓨터의 릴레이에 끼어 있던 실제 나방(bug)을 발견한 데서 유래했다.

**주요 디버깅 전략:**

| 전략 | 설명 |
|:----|:----|
| 오류 메시지 읽기 | 컴파일러 메시지에는 파일명, 줄 번호, 오류 유형이 포함된다 — 항상 주의 깊게 읽을 것 |
| `printf` 디버깅 | 프로그램의 핵심 지점에 `printf`를 삽입하여 변수 값을 출력 |
| 고무 오리 디버깅 | 코드를 한 줄씩 누군가에게 (또는 고무 오리에게) 설명한다 — 설명하는 행위 자체가 버그를 드러내는 경우가 많음 |
| 디버거 사용 | `gdb` (GNU Debugger) 같은 도구로 코드를 단계별 실행하고, 중단점을 설정하고, 변수를 검사 |
| 분할 정복 | 코드의 일부를 주석 처리하여 오류가 발생하는 위치를 좁혀 나감 |

**`printf`를 이용한 기본 디버깅:**

```c
#include <stdio.h>

int main(void) {
    int x = 10, y = 3;
    int result = x / y;

    printf("[DEBUG] x = %d, y = %d\n", x, y);       // 값 확인
    printf("[DEBUG] result = %d\n", result);           // 기대값: 3 (정수 나눗셈)

    // ... 나머지 프로그램
    return 0;
}
```

> **팁:** 코드를 자주 컴파일하는 습관을 들일 것 — 전체 프로그램을 다 작성한 후가 아니라, 몇 줄 작성할 때마다 컴파일한다. 오류를 빨리 잡을수록 수정이 쉬워진다.

> **팁:** 컴파일 시 `gcc -Wall -Wextra`로 모든 경고를 활성화한다. 컴파일러가 프로그램 실행 전에 많은 잠재적 버그를 잡아낼 수 있다.
